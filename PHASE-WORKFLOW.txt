================================================================
STEM QUEST - PHASE WORKFLOW
What to type, session by session, for every phase.
================================================================

The cycle is three sessions, each with one job:

  SESSION 1: PLAN       -> produces a spec
  SESSION 2: IMPLEMENT  -> produces a PR
  SESSION 3 (optional): FIX REVIEW FEEDBACK

Never mix planning and implementing in one session. The /clear
between them is the point: the implementation session starts
fresh and reads only the spec, not a long planning conversation.

There are only FOUR things you ever type. They are marked like
this throughout the file:

  >>> PROMPT ------------------------------------------------
  >>> (what you type into Claude Code)
  >>> -------------------------------------------------------

Everything else is either Claude working or you clicking.


----------------------------------------------------------------
SESSION 1: PLAN THE PHASE
----------------------------------------------------------------

1. Start Claude Code in the repo.

   >>> PROMPT 1 of 4 -----------------------------------------
   >>>
   >>>   /start-phase <phase number or name>
   >>>
   >>>   example:  /start-phase 41
   >>> -------------------------------------------------------

2. Answer the interview questions. Push back where the
   suggestions don't match what you want -- this is the cheapest
   moment in the whole cycle to change direction.

3. When it finishes, OPEN THE SPEC FILE and actually read it:

     docs/specs/phase-<N>-<name>.md

   Check: Is anything in "Out of scope" that you wanted?
   Is anything in scope that you don't? Edit the file directly
   or tell Claude to revise it.

4. When the spec looks right:

   >>> PROMPT 2 of 4 -----------------------------------------
   >>>
   >>>   /clear
   >>>
   >>> -------------------------------------------------------


----------------------------------------------------------------
SESSION 2: IMPLEMENT THE PHASE
----------------------------------------------------------------

5. In the fresh session, paste this exactly, filling in the
   spec path. This is THE implementation prompt -- same wording
   every phase:

   >>> PROMPT 3 of 4 -----------------------------------------
   >>>
   >>>   Read docs/specs/phase-<N>-<name>.md and implement it
   >>>   on a feature branch. Work through the checklist in
   >>>   order. Run /verify-stack before marking anything
   >>>   complete, and /finish-phase when the checklist is
   >>>   done.
   >>>
   >>> -------------------------------------------------------

6. Let it work. Interrupt only if it's going the wrong
   direction -- the spec is the contract, so "that's not in the
   spec" is always a valid correction.

7. /finish-phase runs automatically at the end (it's in the
   prompt above -- you don't type it). It will:
     - verify everything passes
     - check any new migrations with db-migration-checker
     - update the spec checklist and PLAN.md
     - push the branch and open a PR
     - write the handoff
   It will NOT merge. That's yours.


----------------------------------------------------------------
MERGE (you, in the browser / terminal -- no prompts here)
----------------------------------------------------------------

8. Read the PR body. It tells you if there are migrations.

   *** IF THERE ARE NEW MIGRATIONS: APPLY THEM TO NEON ***
   *** BEFORE MERGING. ORDER MATTERS. ***

   From your dev machine (terminal, not Claude):

     cd backend
     DATABASE_URL=<neon connection string> python manage.py migrate

   (Migrations are additive-safe, so old code keeps working
   against the new schema while the deploy rolls out. Merging
   first would deploy new code against the old schema -- that
   direction breaks.)

9. Merge the PR on GitHub. This auto-deploys:
     - backend  -> Render  (stemquest-api.onrender.com)
     - frontend -> Cloudflare Pages

10. After the deploy finishes, check:

     https://stemquest-api.onrender.com/api/health/?deep=1

    Then click through whatever the spec's manual verification
    flow was, on production.


----------------------------------------------------------------
SESSION 3 (ONLY IF NEEDED): FIX REVIEW FEEDBACK
----------------------------------------------------------------

If you reviewed the PR and want changes before merging, start a
fresh session (or continue session 2):

   >>> PROMPT 4 of 4 (only when a PR needs changes) ----------
   >>>
   >>>   Read the open PR for phase <N> (gh pr view --comments)
   >>>   and address the review feedback on the same branch.
   >>>   Run /verify-stack before pushing.
   >>>
   >>> -------------------------------------------------------

Then re-check the PR and go back to the MERGE section.


----------------------------------------------------------------
QUICK REFERENCE: THE FOUR PROMPTS
----------------------------------------------------------------

  1. /start-phase <N>                      (new session)
  2. /clear                                (after approving spec)
  3. Read docs/specs/phase-<N>-<name>.md and implement it on a
     feature branch. Work through the checklist in order. Run
     /verify-stack before marking anything complete, and
     /finish-phase when the checklist is done.
  4. (only if PR needs changes)
     Read the open PR for phase <N> (gh pr view --comments) and
     address the review feedback on the same branch. Run
     /verify-stack before pushing.


----------------------------------------------------------------
RULES THAT KEEP THIS WORKING
----------------------------------------------------------------

- One phase = one branch = one PR. Don't let a phase sprawl.
- Never work on main. /finish-phase checks this, but don't
  start there either -- the implementation prompt says
  "on a feature branch" for a reason.
- Never let Claude merge. Merging = deploying to production.
- If /verify-stack fails, the fix happens BEFORE the PR,
  not after.
- If a session gets long and confused, run /handoff, /clear,
  and re-prompt from the spec. The spec + handoff files are
  the memory; the chat is disposable.
- Backend code changes need: docker compose restart backend
