-------------------------------------------------------------------------------

3a. USE THE BLUEPRINT PATH

Connect Cesar6060/LMS, branch main. It will find render.yaml at the repo root.

  WARNING: do NOT use "New -> Web Service". That path ignores render.yaml
  completely, which means Render won't auto-generate your SECRET_KEY and
  you'll end up filling in every field by hand.

  The tell: if Render ever ASKS YOU for a SECRET_KEY, you're on the wrong
  path. Back out and start over with Blueprint.


3b. FILL IN THE 6 SECRETS

Render will prompt you for exactly these six. SECRET_KEY is deliberately not
on the list - Render creates that one itself.

  DATABASE_URL
      (see the clipboard command below)

  ALLOWED_HOSTS
      stemquest-api.onrender.com

  CSRF_TRUSTED_ORIGINS
      https://stemquest-api.onrender.com

  CORS_ALLOWED_ORIGINS
      http://localhost:5173

  FRONTEND_URL
      http://localhost:5173

  SENTRY_DSN
      (leave completely empty)

The exact formatting matters, and each one breaks differently:

  - ALLOWED_HOSTS is a BARE HOST. No https://, no trailing slash, no path.
    Django compares it to the Host header, which has no scheme.

  - CSRF_TRUSTED_ORIGINS REQUIRES the scheme. Django 4 rejects a bare host
    here. Note this is the exact opposite rule from the line above it.

  - CORS_ALLOWED_ORIGINS needs scheme and port, and NO trailing slash.

  - SENTRY_DSN must be genuinely empty. Not "" and not a placeholder.


To get DATABASE_URL onto your clipboard without it ever showing on screen,
run this and then just paste into Render:

cat /private/tmp/claude-501/-Users-cesar-VideoGameCourseAppCursor-gamedev-platform-v3/7936b940-461c-458b-bfc7-56f905a1f7fa/scratchpad/.neon_url | pbcopy

The string ends in ?sslmode=require&channel_binding=require - keep all of it,
both parameters. SSL is mandatory; the Django settings hardcode it.

LANDMINES - worth reading once before you start
-------------------------------------------------------------------------------

  * populate_java_course is ONE-SHOT in production. Running it a second time
    deletes and rebuilds every unit, which wipes out every student's lesson
    progress and quiz attempts on JAVA101. Harmless today with zero students.
    Destructive the moment there are any.

  * NEVER run seed_data against Neon. It creates six @demo.com accounts with
    a hardcoded password, and its --clear flag deletes EVERY course
    globally, JAVA101 included.

  * ALLOWED_HOSTS being unset does not crash anything. It quietly falls back
    to localhost and then rejects all real traffic with a 400.

  * DEBUG and USE_HTTPS are strict true/false values. A stray space, or
    something like "TRUE!", and the app won't start at all.

  * The deep health check endpoint echoes raw database error text to anyone
    who asks, which can leak host and user details. Not a blocker for this
    phase. Worth narrowing in Phase 40.

  * Media and uploads are knowingly broken in production this phase. Three
    independent causes, all of them Phase 39 work. Not a regression.
